# Ideas Session — 2026-04-25

## Context

User (Xi-Wei Pan, first author of the ProblemReductionPaper) proposes using
`problem-reductions` as the substrate for a new benchmark that measures
AI-agent bug-finding ability. Target venue: NeurIPS Datasets & Benchmarks
track. Rough notes from user:

- "Definition of bug for AI Agent"
- "24 hours find how many bugs"
- "Skills/Agents find counter example"
- "Benchmark/Arena"
- "Extension to benchmark on computation complexity"

## Phase 1 — Exchange 1 (mentor)

**Mentor's take.** The core asset is a machine-checkable oracle:
round-trip preservation + declared polynomial overhead. Unlike SWE-bench and
most code-bug benchmarks, grading can be fully deterministic and needs no
human/LLM judge. This is the pitch-worthy angle.

Three risks raised:

1. **Contamination.** Repo + paper are public. Without held-out or
   injected bugs, scores will be memorization.
2. **Bug supply.** Paper itself claims the library is well-verified;
   real bug density may be too low for a benchmark with discrimination.
3. **Arena vs Benchmark.** NeurIPS D&B prefers reproducible leaderboards
   over open "how-many-in-24h" contests. The two flavors can coexist
   (arena as demo, benchmark as main score) but the main axis needs picking.

Extension to "computation complexity" flagged as out-of-scope for paper 1
(different skill tree: agent-as-theorem-prover).

**Mentor's question back to user:** where does the bug corpus come from?
- (a) historical bugs mined from git log (retrospective, SWE-bench-like)
- (b) injected / mutation-style bugs (synthetic, controllable)
- (c) unknown bugs in a fixed live snapshot
- (d) other / combination

## Phase 1 — Exchange 2 (user)

User picks **(c) with a twist**: fix a benchmark version, and over time
as the library evolves (old bugs fixed, new problems/rules added), the
bug set also evolves — a *living benchmark* model.

## Phase 1 — Exchange 3 (mentor)

Conceded that (a)-historical approach has a counting-granularity problem
(1 fix commit = 1 bug? 3 rules touched = 3 bugs?). Pivoted toward a
mutation-based counterexample-search framing:
- 1 mutant = 1 test item by construction (no counting ambiguity)
- task framed as "find a counterexample" not "find a bug" (oracle is
  deterministic: round-trip pass/fail)
- 3-tier design proposed: Main score (mutation recall) + Taxonomy
  contribution + Living frontier.
- Also offered the "co-verification flywheel" framing.

## Phase 1 — Exchange 4 (user)

User correctly notes the library itself is not provably clean, even with
their harness. I reframed this as a feature, not a bug: round-trip oracle
is one-sided, so "clean baseline" is never needed. Main score stays on
mutants (known positives); bonus score captures counterexamples found in
supposedly-clean rules (genuine wild bugs).

## Phase 1 — Exchange 5 (user — sharp push)

User's key pushback: **"改一处连 unit test 也过不了吧"** — a naive mutant
breaks the test suite, so finding it is trivial. User defines bug as:
**"all tests pass, but latent failure input still exists"**.

This is the test-escaping / stubborn-mutant concept from Jia & Harman 2011
mutation testing survey. Reframed benchmark around a 2-stage filter:
generate mutants → drop those caught by test suite → confirm remainder
are truly buggy (not equivalent) via held-out round-trip check. What
remains = benchmark test set of "test-escaping bugs".

Three secondary wins noted:
- Self-consistent with Paper 1 narrative (harness is good, yet these bugs
  exist — benchmark measures what's left).
- Taxonomy of test-escaping bug patterns in reduction code is itself a
  contribution.
- Flywheel: found counterexamples become new unit tests, pushing the
  bug-discovery frontier harder each cycle.

Open question flagged: Stage-1 generation strategy — LLM vs handcrafted
operator vs higher-order mutation.

## Phase 1 — Exchange 6 (user) and Mentor response

User asks: **"单纯就比找到 bug 的数量不行吗"**. Mentor concedes
over-engineering, repositions the proposal as **Arena**, not Benchmark
(matching user's original "Benchmark/Arena" note). Recommends `by-rule`
dedup for cleanest leaderboard semantics. Distinguishes Benchmark
tradition (MMLU/SWE-bench, fixed denominator) from Arena tradition
(Chatbot Arena/Pwn2Own/MLE-bench, count-based, NeurIPS-acceptable).

## Phase 1 — Exchange 7 (user) — pilot planning

User picks `by-rule` dedup; admits not knowing latent-bug density;
requests pilot plan. Mentor delivers full pilot plan:
- Freeze snapshot (commit SHA)
- Build oracle as `problemreductions-cli verify` subcommand
- Sample ~20 rules across (suspicious / transitive / random)
- 3 agents × 20 rules × 4h compressed budget
- Decision gate at N rules-with-counterexample: ≥8 green, 3-7 yellow,
  ≤2 red (pivot to mutation-injected design)

Quick scan of submodule confirmed: `src/rules/*.rs` (per-rule files),
`src/unit_tests/rules/`, existing CLI to extend. Naming question raised
("pr-oracle" was a placeholder; fold into existing CLI).

## Phase 1 — Exchange 8 (user) — capability framing

User asks: **what AI agent capability does this benchmark reflect — we
need motivation**. Mentor crystallizes the motivation as
**"Test-Escaping Bug Discovery"** (primary), with two alternative
framings as supporting layers:

- (A) **Test-Escaping Bug Discovery** — primary; positions benchmark as
  complementary to SWE-bench. Agent must find bugs that pass-test-driven
  evaluation cannot detect. Predicted headline finding: SOTA agents with
  >60% SWE-bench score will achieve <20% on this benchmark.
- (B) **Adversarial reasoning on formal artifacts** — philosophical
  layer; reductions as miniature math claims, counterexample search as
  constructive falsification.
- (C) **Agentic QA / independent verification** — industrial relevance
  layer.

Operational implication noted: pilot scoring must distinguish "bug on
test-covered code path" (true test-escaping) from "bug on uncovered
path" (mere coverage gap). Suggested adding `cargo tarpaulin`-style
coverage tagging to pilot.
