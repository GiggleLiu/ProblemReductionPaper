# Ideas Session — 2026-04-25

## Context

User (Xi-Wei Pan, first author of the ProblemReductionPaper) proposes using
`problem-reductions` as the substrate for a new benchmark that measures
AI-agent bug-finding ability. Target venue: NeurIPS Datasets & Benchmarks
track. Rough notes from user:

- "Definition of bug for AI Agent"
- "24 hours find how many bugs"
- "Skills/Agents find counter example"
- "Benchmark/Arena"
- "Extension to benchmark on computation complexity"

## Phase 1 — Exchange 1 (mentor)

**Mentor's take.** The core asset is a machine-checkable oracle:
round-trip preservation + declared polynomial overhead. Unlike SWE-bench and
most code-bug benchmarks, grading can be fully deterministic and needs no
human/LLM judge. This is the pitch-worthy angle.

Three risks raised:

1. **Contamination.** Repo + paper are public. Without held-out or
   injected bugs, scores will be memorization.
2. **Bug supply.** Paper itself claims the library is well-verified;
   real bug density may be too low for a benchmark with discrimination.
3. **Arena vs Benchmark.** NeurIPS D&B prefers reproducible leaderboards
   over open "how-many-in-24h" contests. The two flavors can coexist
   (arena as demo, benchmark as main score) but the main axis needs picking.

Extension to "computation complexity" flagged as out-of-scope for paper 1
(different skill tree: agent-as-theorem-prover).

**Mentor's question back to user:** where does the bug corpus come from?
- (a) historical bugs mined from git log (retrospective, SWE-bench-like)
- (b) injected / mutation-style bugs (synthetic, controllable)
- (c) unknown bugs in a fixed live snapshot
- (d) other / combination

## Phase 1 — Exchange 2 (user)

User picks **(c) with a twist**: fix a benchmark version, and over time
as the library evolves (old bugs fixed, new problems/rules added), the
bug set also evolves — a *living benchmark* model.
