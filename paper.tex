\title{Agentic Assembly of Computational Hard Problems}

\author{...}  % placeholder

\maketitle

\begin{abstract}
A unified library of polynomial-time reductions between NP-hard problems would let practitioners route any supported problem to a specialized solver---quantum hardware, commercial optimizers, or domain heuristics---through a single interface.
No such library exists, because building one by hand demands a shared formalism across many contributors and ongoing maintenance as the literature grows.
We address this gap with AI coding agents steered by \emph{skills}: composable workflow documents that encode project conventions and domain knowledge.
The approach yields a library of over 100 problem types, due to three changes.
(1)~A \emph{no-code contribution route} lets domain experts add reductions by filing structured issues; no familiarity with the implementation language is needed.
(2)~A \emph{multilayer verification stack} checks each reduction from type signatures through property-based tests to \emph{agentic feature tests}---automated agents that act as virtual users, exercising the library end-to-end.
(3)~A \emph{fully automated pipeline} lets one maintainer run the project, and a new maintainer can on-board in half a day.
The entire stack was built in ten weeks. The methodology shows that agents, given the right scaffolding, can produce verified software artifacts whose scope would be impractical to maintain by hand.
\end{abstract}

\section{Introduction}\label{sec:intro}

\subsection{The Problem: Many Hard Problems, Few Solvers}

Combinatorial optimization problems arise throughout science and engineering.
An airline needs to assign crews to flights.
A chip designer needs to allocate registers.
A logistics company needs to route delivery trucks.
Each of these is an instance of an NP-hard problem---a class of problems for which no efficient general-purpose algorithm is known, but which can be solved in practice for moderate sizes by specialized solvers.

The difficulty is that each solver speaks its own narrow language.
Rydberg atom arrays~\cite{lucas2014, pichler2018}---a type of quantum hardware---natively solve the Maximum Independent Set problem on geometric graphs.
D-Wave quantum annealers~\cite{glover2019} solve Quadratic Unconstrained Binary Optimization (QUBO).
Commercial engines like Gurobi and CPLEX solve Integer Linear Programs.
Physics-inspired methods scale further---graph neural networks solve QUBO at million-variable scale~\cite{Schuetz2022PhysicsGNN}, and hybrid quantum-GNN methods tackle the Traveling Salesman Problem~\cite{He2024QuantumTSP}---but still require a QUBO or Ising formulation as input.
A practitioner with a graph coloring problem cannot directly use any of these solvers without first \emph{translating} the problem into a form the solver understands.

This translation is called a \emph{reduction}: a polynomial-time algorithm that converts an instance of one problem into an instance of another, together with an inverse map that translates the solution back.
Reductions are the central tool of computational complexity theory~\cite{karp1972}, but they also have immediate practical value: each verified reduction is a bridge connecting a new problem to an existing solver.

\subsection{Bridging Problems to Solvers}

A single reduction between two problems is a \emph{primitive reduction rule}.
To bridge \emph{many} problems to \emph{many} solvers, we organize primitive rules as a \emph{directed graph}.
Each node is a problem type; each directed edge is a primitive rule.
Every edge carries a \emph{reduction overhead}---a polynomial mapping the source's size measures to the target's.
\begin{center}
  \includegraphics[width=0.70\columnwidth]{figures/reduction-graph-concept.pdf}
\end{center}
Reductions compose along directed paths.
Chaining $A \to B$ and $B \to C$ yields $A \to C$.
Writing $r_{X \leftarrow Y}$ for the overhead of rule $Y \to X$, the composite overhead is $r_{C \leftarrow A} = r_{C \leftarrow B} \circ r_{B \leftarrow A}$.
When multiple paths connect a problem to different solvers, the system compares total cost---reduction overhead plus solver complexity---and selects the cheapest route for a given input size.
Adding a primitive rule strengthens the graph only when it improves on existing routes: a direct $A \to C$ with provably lower overhead than the composite $A \to B \to C$ earns its place; one that does not is omitted to keep path-finding tractable.

Two kinds of paths play a special role.
A path from well established NP-complete problems like 3-SAT to a problem~$H$ constitutes an NP-hardness proof for~$H$---we say $H$ is \emph{NP-hard verified} by the graph.
A path from a problem to integer linear programming (ILP) grants access to efficient solvers (HiGHS, Gurobi, CPLEX, etc.), providing a verification oracle for testing reductions end-to-end.

\subsection{The Challenge}

Building such a graph is itself a hard problem---not computationally, but organizationally.
Earlier attempts illustrate the difficulty.
QUBO-specific tools---PyQUBO~\cite{Zaman2021PyQUBO}, qubovert~\cite{Iosue2022qubovert}, ToQUBO.jl~\cite{Monteiro2022ToQUBO}, and D-Wave's Ocean SDK~\cite{DWaveOcean2023}---convert problems into a single target formulation but provide no inter-problem reductions and no inverse maps for solution extraction.
Lucas~\cite{lucas2014} catalogued Ising formulations for Karp's 21~problems, but the result is a paper, not composable code.
The Karp DSL~\cite{Zhang2022Karp} and REDNP~\cite{Creus2014REDNP} provide languages for writing and testing reductions, but are pedagogical tools with no solver integration.
ProblemReductions.jl~\cite{Liu2024ProblemReductions}, the most direct predecessor, implements 18~problem types in Julia with a reduction graph structure---but has remained small (3~contributors, 13~stars) after two years.
The pattern is consistent: each project covers a handful of problems, then stalls.
More broadly, the gap between \emph{known} and \emph{implemented} reductions has widened for decades.
Hundreds of reductions exist in the literature---Garey and Johnson~\cite{garey1979} alone catalogue over 300~NP-hard problems with reduction sketches---yet the largest software implementation covers fewer than 20.
The bottleneck was never mathematical knowledge; it was building and maintaining the software.

The graph's value grows with every edge, but so does the cost of maintaining it.
Every edge must conform to the same conventions---same interface, same file layout, same test pattern---because this uniformity is what makes edges composable.
Maintaining such uniformity across many contributions is precisely what human teams cannot do at scale.
Conventions drift as contributors accumulate their own patterns (\emph{convention drift}).
The domain experts who know the reductions rarely write systems code, bottlenecking progress on the few who know both (\emph{knowledge discontinuity}).
When a new reduction appears in the literature, the mathematician who understands it has no way to contribute it without learning the implementation language and the project's conventions.
And the repetitive verification cycle that each edge demands---coding, testing, documenting---exhausts even dedicated maintainers (\emph{effort exhaustion}).
Meanwhile, systems like FunSearch~\cite{RomeraParedes2023FunSearch} and AlphaEvolve~\cite{Novikov2025AlphaEvolve} can \emph{discover} novel reductions through evolutionary search, and URSA~\cite{Janicic2025URSA} can verify them via SAT-based constraint solving---but discovered reductions still need to be implemented, tested, and integrated into a composable library.

AI coding agents offer a way forward.
The evolution from SWE-agent~\cite{Yang2024SWEagent} and Devin~\cite{Wu2024Devin} to OpenHands~\cite{Wang2024OpenHands} and Claude Code~\cite{Anthropic2025ClaudeCode} has expanded single-task capabilities to 70--80\% on SWE-Bench~\cite{Xia2025LiveSWEagent}, but longer-horizon benchmarks reveal a capability cliff~\cite{Thai2025SWEEVO, Deng2025SWEBenchPro}, and Roychoudhury~\cite{Roychoudhury2025AgenticAI} identifies specification inference as the central difficulty.
Full formal verification is also not yet practical at this scale: VeriCoding~\cite{Bursuc2025VeriCoding} reports 27--44\% success on formal specifications, and VeriBench~\cite{Miranda2025VeriBench} finds agents approach 90\% compilation in Lean~4 but rarely complete proofs.
The question is whether agents, given the right scaffolding, can produce \emph{verified} software at a scale that human teams have not achieved.
The scaffolding we propose is \emph{skills}.
A skill is a plain-markdown document, committed to the repository, that reads like a checklist: ``implement the forward and inverse maps, write round-trip tests, register the rule in the CLI, draft a paper entry with a proof sketch.''
The agent interprets each step using its own capabilities, but the skill ensures nothing is skipped and every step follows the project's conventions.
A \emph{project specification} (\texttt{CLAUDE.md} or \texttt{AGENTS.md}) anchors the system as a single source of truth for architecture and constraints that every skill inherits.
Skills address all three barriers: they eliminate convention drift by enforcing the same interface, layout, and test pattern at every invocation; they bridge knowledge discontinuity by letting domain experts contribute through structured issues while agents handle the implementation; and they absorb effort exhaustion by making the routine work---coding, testing, documenting---repeatable without human attention.

We call this an \emph{assembly problem}: software whose subtasks are homogeneous and formally verifiable, but whose scale exceeds what human teams can sustain.
Reduction-graph construction is a natural instance: every reduction admits a \emph{round-trip test}---reduce a small instance, solve the target problem, map the solution back, and check optimality---without a problem-specific oracle.
This cheap, automatic verification makes the task well suited for AI coding agents, which produce volume while round-trip tests catch the errors.
A domain expert who discovers a new reduction contributes it by filing a structured issue---no code required---and agents handle the implementation, testing, and documentation.
Skills enforce project conventions at every invocation, and a multilayer verification stack rejects non-conforming code before it enters the repository.

We demonstrate that AI coding agents, constrained by systematic verification, can build mathematical software at a scale that human teams have not achieved.
The evidence is a Rust library of 27~problem types (over 100~variants when parameterized by graph and weight type) and 45~reduction rules with $>$95\% test coverage, built in nine weeks by a single maintainer and AI agents.
Three ideas make this possible:
\begin{enumerate}
  \item A \emph{skill-based automation pipeline} built on two kinds of skills: \emph{implementation skills} that let agents autonomously code, test, and document each reduction, and \emph{human-facing skills} that guide users, contributors, and maintainers through the project's conventions (\Cref{sec:skills}).
  \item A \emph{no-code contribution route} that lets domain experts continuously integrate new reduction rules by filing structured issues---no familiarity with the implementation language is needed (\Cref{sec:method}).
  \item A \emph{multilayer verification stack} that ensures correctness at every stage: automated unit tests and round-trip tests validate each reduction programmatically, while domain experts perform end-to-end checks against contributor-specified ground truth (\Cref{sec:verification}).
\end{enumerate}

The \texttt{pred} CLI is the primary entry point.
Consider a logistics engineer who models warehouse bin placement as a Maximum Independent Set problem on a conflict graph: vertices are candidate locations, edges connect locations that are too close.

Lines~3--5: \texttt{pred create} builds a problem instance from a conflict graph, \texttt{pred reduce} converts it to ILP along the cheapest path, and \texttt{pred solve} solves the resulting ILP and maps the solution back to the original problem.
The output (lines~6--11) shows that locations~0 and~3 form a maximum independent set of size~2.

Beyond this workflow, \texttt{pred path} shows the reduction route and per-step overhead between any two problem types, \texttt{pred show} prints a problem's definition and neighboring reductions, and \texttt{pred list} shows all registered problem types with their complexities.
The same functionality is available through the Rust library API.
The build system also produces an auto-compiled PDF manual containing formal definitions, reduction theorems with proof sketches, and worked examples for every implemented model and rule.

\subsection{Core Library}\label{sec:graph-structure}

The core crate contains six modules (\Cref{fig:library-architecture}, middle).

\emph{Problem types.}
Every problem type implements a uniform evaluation interface: given any configuration, return a score (\Cref{app:architecture}).
Each type defines its own \emph{size measures} and \emph{best-known algorithm complexity}---for example, Maximum Independent Set (MIS) is parameterized by $|V|$ and $|E|$, with a known $O(1.1996^{|V|})$ algorithm.
The uniform evaluation interface enables brute-force solving on small instances, which powers round-trip testing.

\emph{Reduction rules.}
Each rule requires both a forward map and an inverse map in a single implementation---an agent cannot compile a forward reduction without providing solution extraction (\Cref{app:architecture}).
Each rule carries a \emph{reduction overhead}: a multivariate polynomial mapping the source's size measures to the target's.
For example, the Satisfiability $\to$ MIS reduction maps a formula with $n_l$ literals to a graph with $O(n_l)$ vertices and $O(n_l^2)$ edges.

\emph{Graph engine.}
Problem types and reduction rules compose into a directed graph.
To route a problem instance to a solver, the engine finds the cheapest reduction path using Dijkstra's algorithm.
The cost function is configurable: minimize the number of reduction steps, or minimize a specific output dimension such as \texttt{num\_vars}.

\emph{Symbolic engine.}
Both reduction overhead and algorithm complexity are represented as symbolic mathematical expressions.
The symbolic engine supports three operations: \emph{composition} (substituting one edge's output into the next edge's input), \emph{comparison} (checking whether a composite path dominates a direct rule, for redundancy detection), and \emph{evaluation} (computing numeric cost for a concrete instance size, enabling instance-dependent path search).

\emph{Solvers.}
A brute-force enumerator exhaustively searches small instances---used in round-trip testing where correctness, not speed, matters.
An ILP solver handles larger instances; the backend is swappable, defaulting to the open-source HiGHS~\cite{Huangfu2018HiGHS} with Coin-CBC, SCIP, and others as drop-in alternatives.
Most problem types have a reduction path to ILP, directly or through intermediate types.

\emph{Example database.}
Canonical instances---extracted from contributor issues---serve as the single source of truth.
The same instance feeds round-trip tests, JSON fixtures rendered in the PDF manual, and the \texttt{pred create --example} command for interactive exploration.

\subsection{Why Rust?}\label{sec:why-rust}

The choice of implementation language has outsized impact in agentic workflows, because the agent's edit--compile--test loop runs hundreds of times per session.
None of the maintainers had written Rust before this project---the language was chosen for properties that benefit agents, not developers.

\emph{Explicit, actionable error messages.}
The Rust compiler produces diagnostics that include the error location, the conflicting types or lifetimes, and often a suggested fix.
Agents parse these messages and resolve errors without human intervention---a property we rely on throughout the pipeline.
Languages with less structured diagnostics (e.g., C++ template errors) would require more agent reasoning per cycle.

\emph{Fast feedback.}
Incremental compilation and the built-in test harness produce results in seconds.
A typical round-trip test compiles and runs in under 3~seconds, enabling agents to iterate rapidly.
High feedback rate---short cycles between edit and result---is the single most important factor in agent productivity.

Rust's well-known strengths---memory safety eliminating entire bug categories at compile time, high performance, a 7\,MB binary, and Cargo's integrated toolchain---further reduce the surface area of errors agents must debug.

\section{Skills and Automation}\label{sec:method}

\subsection{The Automation Pipeline}\label{sec:pipeline-overview}

Adding a reduction to the graph involves five stages, from initial proposal to merged code (\Cref{fig:pipeline}).
Not all stages are equal: some are routine work that an agent can execute autonomously; others require creativity and domain judgment that no instruction can encode.

\begin{figure*}[t]
  \centering
  \includegraphics[width=\textwidth]{figures/pipeline.pdf}
  \caption{Contribution pipeline mirroring the GitHub project board.
    Four roles drive the flow: a \emph{domain expert} files issues (orange), a \emph{maintainer} triages and merges (orange), an \emph{implementation agent} picks issues and creates PRs (blue), and a \emph{review agent} runs parallel reviews and posts verdicts (teal).
    Review Pool is the handoff boundary between the two agents.
    Failures move items to On~Hold for human triage.}
  \label{fig:pipeline}
\end{figure*}

\textbf{Stage~1: Propose.}
A domain expert contributes the mathematical content: which problem to add, what its formal definition is, what reduction algorithm connects it to existing problems.
The \texttt{propose} skill supports this process through a structured brainstorming session---one question at a time, using mathematical language only, no programming required.
The agent assists by running a topology analysis to identify the most impactful gaps---problems unreachable from 3-SAT (lacking NP-hardness proofs), orphan nodes with no edges, clusters disconnected from solvers---and recommends high-value rules the contributor may not have considered.
It also checks for conflicts with existing issues and duplicate proposals.
Before filing, the agent pre-validates the draft against Stage~2's quality checks; only issues passing all four checks are submitted.

\textbf{Stage~2: Validate.}
The \texttt{check-issue} skill automatically flags potential problems along four dimensions: \emph{usefulness} (does a cheaper path already exist?), \emph{non-triviality} (is this a genuine structural transformation?), \emph{correctness} (do cited references support the claims?), and \emph{writing quality} (are all symbols defined, all examples fully worked?).
It posts a structured report as a GitHub comment and labels the issue accordingly.
The maintainer then uses the \texttt{fix-issue} skill to resolve flagged uncertainties: it auto-corrects mechanical problems (undefined symbols, missing metrics) and brainstorms substantive fixes interactively.
Only issues passing all four checks move to Ready.

\textbf{Stage~3: Implement.}
The \texttt{run-pipeline} skill picks the highest-ranked Ready issue, claims it, and orchestrates three layers of skills.
First, domain skills (\texttt{add-model}, \texttt{add-rule}) encode the conventions for adding a problem type or reduction rule correctly---Rust types, trait implementations, unit tests, CLI registration, and a paper entry with formal definitions and proof sketches.
Second, software engineering skills (\texttt{issue-to-pr}) encode PR best practices: interpret the issue, create an isolated git worktree, write an implementation plan, execute it, and open a PR with a detailed summary.
Third, a sub-agent orchestration layer dispatches independent tasks to parallel agents, powered by the Superpowers plugin~\cite{Obra2025Superpowers}---a general-purpose toolkit for structured planning, worktree isolation, and sub-agent-driven development.
Each PR contains exactly one model or one rule.
Fully automatable.

\textbf{Stage~4: Review.}
The \texttt{review-pipeline} skill dispatches three read-only sub-agents in fresh context windows---structural completeness, code quality, and agentic feature testing (\Cref{sec:verification})---then posts a combined verdict as a PR comment.
Fresh context prevents confirmation bias.
Fully automatable.

\textbf{Stage~5: Merge.}
The \texttt{final-review} skill automatically fixes mechanical issues (formatting, naming conventions, missing registrations) and presents uncertain fixes to the maintainer for decision.
It then walks the maintainer through the agentic verdict, assesses usefulness, safety, and completeness, and either merges or holds the PR with a reason.

\textbf{Stage~6: Verify.}
The maintainer invites the domain expert who filed the original issue to a community call.
Together they walk through the implementation visually and interactively---inspecting the formal definition, the canonical example, and the proof sketch in the compiled PDF manual.
This closing-the-loop step catches errors invisible to automated tests: subtle misinterpretations of the problem statement or proof arguments that are formally correct but mathematically unconvincing.

Stages~3--4 (implement and review) are fully automatable and run in \emph{headless mode}---no human interaction---via OpenAI Codex~\cite{OpenAI2025Codex}; the human-facing stages use Claude Code~\cite{Anthropic2025ClaudeCode}, whose interactive terminal supports the dialogue that the remaining stages require.

\subsection{Skill Design}\label{sec:skills}

Building a verified reduction library requires both mathematical judgment and routine engineering.
We found that four responsibilities must remain with humans: \emph{selecting} which reduction to pursue (understanding the need), \emph{verifying} the algorithm's correctness (requires deep reasoning), \emph{constructing} canonical examples (creative work), and \emph{authorizing} merges (security critical).
Current models cannot substitute for any of these: reasoning degrades sharply with problem complexity~\cite{Mirzadeh2025GSMSymbolic,Shojaee2025IllusionOfThinking,Dziri2023FaithFate}, research-level mathematics remains largely unsolved~\cite{Glazer2024FrontierMath,Paster2025HLE,Merrill2024ExpressivePower}, and unsupervised merge access is a security risk.

We encode these boundaries in skills---workflow documents that define \emph{what} must happen without fixing \emph{how}.
We classify the eighteen skills into two types (\Cref{fig:skill-architecture}): \emph{advisor skills} (blue) pause mid-execution to consult a human for mathematical or security decisions; \emph{agent skills} (white) run fully autonomously.
A single \texttt{run-pipeline} invocation triggers the full cascade from issue to pull request; the maintainer's effort scales with the number of \emph{skill types}, not the number of reductions.

\subsection{Correctness by Construction}\label{sec:verification}

Correctness begins with the issue, not the code.
The structured issue serves as both the specification and the acceptance criterion: it defines the reduction's mathematical content, provides the canonical example that seeds all tests, and sets the quality bar through explicit completeness checks.

\textbf{Issue check.}
Before any code is written, the proposal itself is validated along four dimensions: \emph{usefulness} (does a cheaper path already exist?), \emph{non-triviality} (is this a genuine structural transformation, not a trivial renaming?), \emph{correctness} (do cited references support the claims?), and \emph{writing quality} (are all symbols defined, all examples fully worked?).
Invalid proposals are rejected before implementation effort is spent.
When a contributor batch-submitted 322~issues, only 25\% passed all four checks; the rest were incomplete specifications~(39\%), factually incorrect~(20\%), trivial~(13\%), or without practical value~(6\%).
The issue's canonical example---a small, fully worked instance with known optimal solution---later seeds the round-trip test, the PDF manual's visual diagram, and the CLI's \texttt{-{}-example} mode.
The contributor's mathematical intent propagates unchanged through all downstream artifacts (\Cref{fig:roundtrip-pipeline}).

    The contributor cross-checks the manual against the original issue (dashed arrow).}

\textbf{Agentic feature tests.}
Open-source libraries normally stabilize through years of community use: real users file bug reports, request clarifications, and expose edge cases that developers never anticipated.
A niche library cannot wait for that community to form.
Agentic feature tests compress this feedback loop from years to minutes by \emph{simulating} the community.

The mechanism has three phases.
First, the main agent launches a sub-agent in a \emph{fresh context window}---one that has never seen the implementation---and injects an \emph{agent profile}: a structured persona with domain expertise, testing habits, and a concrete use case (\Cref{fig:agent-profile}).
The fresh context is essential; a sub-agent that has just written the code cannot objectively test it.
Second, the sub-agent role-plays the persona end-to-end: it reads only the project's documentation, picks a problem it recognizes from its domain, builds an instance with realistic data, discovers a reduction path through the CLI, solves, and verifies the result against its own expectation.
Third, the sub-agent steps out of character.
The main agent interviews it---what worked, what was confusing, where did the output diverge from expectation---and the sub-agent writes a structured problem report.
This report surfaces usability gaps, documentation omissions, and semantic errors that unit tests cannot reach.

\section{Discussion and Conclusion}\label{sec:conclusion}

\subsection{Discussion}

The methodology generalizes to any \emph{gallery library}---a collection of homogeneous, independently verifiable implementations with uniform interfaces.
An automatic differentiation (AD) engine is a natural example: each differentiation rule transforms one operation into its derivative, and finite-difference checks play the same role as round-trip tests---verifying correctness automatically, without a problem-specific oracle.
The key requirement is a cheap, generic verification mechanism that applies uniformly across all entries.
The cost of entry is skill engineering: the 18~skills in this project represent substantial upfront investment, and each new domain requires its own set.
The methodology does \emph{not} generalize to heterogeneous tasks---the staple of SWE-Bench---where each issue is structurally unique and resists skill-based decomposition.

The library itself can serve as a \emph{tool} for AI agents: an agent facing an optimization problem can invoke the \texttt{pred} CLI---or its Model Context Protocol (MCP) server---to route the problem through the reduction graph to a solver, without knowing which solver fits or how to formulate the reduction.
As the graph grows, so does the set of problems any MCP-connected agent can solve.
The reduction graph also serves as a \emph{solver-agnostic compilation layer}: adding a single edge from a hardware platform's native formulation (e.g., MIS for Rydberg atoms) routes all supported problem types to that device.

Three directions extend this work: composing with \emph{automated discovery} via evolutionary search~\cite{Novikov2025AlphaEvolve}, supplementing round-trip tests with \emph{formal verification} (machine-checked Lean or Coq proofs), and \emph{cost-aware path selection} where the optimal solver route depends on instance scale.

\subsection{Conclusion}

We have introduced \emph{assembly problems}---software projects whose scale exceeds human capacity but whose homogeneous, verifiable structure makes them amenable to agent execution constrained by systematic verification.
NP-hard problem reductions are a concrete instance: over nine weeks, a single maintainer and AI agents produced a verified reduction graph connecting 27~problem types to specialized solvers.

Three structural barriers---convention drift, effort exhaustion, and knowledge discontinuity---make assembly problems infeasible for human teams.
Verification ensures that agents cannot deviate from contributor-specified ground truth, and skills encode workflow knowledge as reusable documents, lowering the contribution barrier from ``knows the programming language'' to ``knows the mathematics.''
As the graph scales toward 100+ problem types, it becomes a reduction compiler---routing any supported problem to any connected solver automatically.

\section{System Architecture}\label{app:architecture}

The library's type system reduces the space of possible agent errors by making incorrect code fail to compile.
This appendix describes four mechanisms---the \texttt{Problem} trait, the \texttt{ReduceTo} trait, the \texttt{\#[reduction(overhead)]} macro, and the \texttt{declare\_variants!} registry---that enforce correctness structurally (\Cref{fig:architecture}).

\begin{figure}[t]
  \centering
  \includegraphics[width=\columnwidth]{figures/architecture.pdf}
  \caption{Trait hierarchy and compile-time validation.
    The \texttt{Problem} trait defines a universal evaluation interface; \texttt{ReduceTo<T>} requires both forward and inverse maps; procedural macros validate overhead expressions and variant registrations at compile time.}
  \label{fig:architecture}
\end{figure}

\paragraph{The Problem trait.}
Every problem type implements \texttt{Problem}, which requires: a constant name, an associated metric type (\texttt{SolutionSize<W>} for optimization problems, \texttt{bool} for decision problems), a method returning the configuration space dimensions, an \texttt{evaluate()} method that scores any configuration, and variant metadata for type-parameter tracking.
The key member is \texttt{evaluate()}: because every problem maps configurations to metrics through this uniform interface, a brute-force solver can enumerate the space and select the best solution, enabling round-trip testing without problem-specific oracles.

\paragraph{The ReduceTo trait.}
The generic \texttt{ReduceTo<T>} trait requires a \texttt{reduce\_to()} method returning a \texttt{ReductionResult} that bundles \texttt{target\_problem()} and \texttt{extract\_solution()}.
By requiring both forward and inverse in a single implementation, the type system ensures every reduction is round-trip capable---an agent cannot compile a forward reduction without providing the extraction logic.

\paragraph{Compile-time overhead validation.}
The \texttt{\#[reduction(overhead = \{...\})]} procedural macro attaches symbolic size expressions to each reduction.
These expressions are parsed at compile time; variable names are validated against getter methods on the source type, so a typo (e.g., referencing \texttt{num\_vertex} when the method is \texttt{num\_vertices}) causes a compile error rather than a silent bug.

\paragraph{Variant registry.}
Problem types are parameterized by graph type (SimpleGraph, KingsSubgraph, etc.) and weight type (unit weight, \texttt{i32}, \texttt{f64}).
The \texttt{declare\_variants!} macro registers concrete instantiations with their best-known complexity, enabling automated graph export, documentation completeness checking, and redundancy analysis.
