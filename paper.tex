\title{Agentic Assembly of Computational Hard Problems}

\author{...}  % placeholder

\maketitle

\begin{abstract}
A unified library of polynomial-time reductions between NP-hard problems would let practitioners route any supported problem to a specialized solver through a single interface.
Target solvers range from quantum hardware to commercial optimizers and domain heuristics.
No such library exists: every reduction must conform to a shared interface, but conventions drift across contributors, the domain experts who know the reductions rarely write systems code, and the repetitive verification cycle exhausts maintainers.
We address this gap with AI coding agents steered by \emph{skills}: plain-text checklists, committed to the repository, that encode the project's conventions and domain knowledge.
Three design decisions make this scale possible.
(1)~A \emph{no-code contribution route} lets domain experts add reductions by filing structured issues.
(2)~A \emph{multilayer verification stack} checks each reduction from type signatures through property-based tests to end-to-end \emph{agentic feature tests}---automated agents that exercise the library as virtual users.
(3)~A \emph{fully automated pipeline} lets a small team run the entire project.
The library reached 100+ problem types in under three months of development.
The results suggest that agents, given the right scaffolding, can produce verified software artifacts at a scale that human teams have not achieved.
\end{abstract}

\section{Introduction}\label{sec:intro}

\subsection{The Problem: Many Hard Problems, Few Solvers}

Combinatorial optimization problems arise throughout science and engineering.
An airline needs to assign crews to flights.
A chip designer needs to allocate registers.
A logistics company needs to route delivery trucks.
Each of these is an instance of an NP-hard problem---a class of problems for which no efficient general-purpose algorithm is known, but which can be solved in practice for moderate sizes by specialized solvers.

The difficulty is that each solver speaks its own narrow language.
Rydberg atom arrays~\cite{lucas2014, pichler2018}---a type of quantum hardware---natively solve the Maximum Independent Set problem on geometric graphs.
D-Wave quantum annealers~\cite{glover2019} solve Quadratic Unconstrained Binary Optimization (QUBO).
Commercial engines like CPLEX~\cite{IBM2022CPLEX} solve Integer Linear Programs.
Physics-inspired methods scale further---graph neural networks solve QUBO at million-variable scale~\cite{Schuetz2022PhysicsGNN}, and hybrid quantum-GNN methods tackle the Traveling Salesman Problem~\cite{He2024QuantumTSP}---but still require a QUBO or Ising formulation as input.
A practitioner with a crew-scheduling problem cannot reach most of these solvers without first \emph{translating} the problem into a form the solver understands.

This translation is called a \emph{reduction}: an efficient algorithm that converts an instance of one problem into an instance of another, together with an inverse map that translates the solution back.
Borrowing the idea of reductions from complexity theory~\cite{karp1972}, we require each reduction to carry an inverse map that translates the solution back.
Each verified reduction is a bridge connecting a new problem to an existing solver.

\subsection{Bridging Problems to Solvers}

A single reduction between two problems is a \emph{primitive reduction rule}.
To bridge \emph{many} problems to \emph{many} solvers, we organize primitive rules as a \emph{directed graph}.
Each node is a problem type; each directed edge is a primitive rule.
Every edge carries a \emph{reduction overhead}---a polynomial mapping the source's size measures to the target's.
The figure below illustrates the structure:
\begin{center}
  \includegraphics[width=0.70\columnwidth]{figures/reduction-graph-concept.pdf}
\end{center}
Reductions compose along directed paths.
Chaining $A \to B$ and $B \to C$ yields $A \to C$.
Each overhead is a function from input size to output size; composing overheads means composing these functions.
Writing $r_{X \leftarrow Y}$ for the overhead of rule $Y \to X$, the composite overhead is $r_{C \leftarrow A} = r_{C \leftarrow B} \circ r_{B \leftarrow A}$.
When multiple paths connect a problem to different solvers, the system selects the cheapest route for a given input size.

Two kinds of paths play a special role.
A path from a well-established NP-complete problem like 3-SAT to a problem~$H$ provides an NP-hardness argument for~$H$, backed by tested code rather than a pen-and-paper proof---we say $H$ is \emph{NP-hard verified} by the graph.
A path from a problem to integer linear programming (ILP) grants access to mature solvers (HiGHS, Gurobi, CPLEX, etc.).

\subsection{The Challenge}

Building such a graph is itself a hard problem---not computationally, but organizationally.
Despite decades of work on reduction libraries and QUBO tools (\Cref{sec:related}), the gap between \emph{known} and \emph{implemented} reductions has only widened.
Garey and Johnson~\cite{garey1979} catalogue over 300~NP-hard problems with reduction sketches, yet the largest software implementation covers fewer than 20.
The bottleneck was never mathematical knowledge; it was building and maintaining the software.

The graph's value grows with every edge, but so does the cost of maintaining it.
Every edge must conform to the same conventions---same interface, same file layout, same test pattern---because this uniformity is what makes edges composable.
Maintaining such uniformity across many contributions is precisely what human teams cannot do at scale.
Conventions drift as contributors accumulate their own patterns (\emph{convention drift}).
The domain experts who know the reductions rarely write systems code, bottlenecking progress on the few who know both (\emph{knowledge discontinuity}).
When a new reduction appears in the literature, the mathematician who understands it has no way to contribute it without learning the implementation language and the project's conventions.
And the repetitive verification cycle that each edge demands---coding, testing, documenting---exhausts even dedicated maintainers (\emph{effort exhaustion}).

We call this an \emph{assembly problem}: a software project whose subtasks share the same interface and admit a cheap, generic correctness test (here, round-trip testing), but whose count exceeds what human teams can sustain.
Reduction-graph construction is a natural instance: every reduction admits a \emph{round-trip test}---reduce a small instance, solve the target problem, map the solution back, and check optimality---without a problem-specific oracle.
AI coding agents offer a way forward (\Cref{sec:related}).
Assembly problems sidestep the capability cliff reported on longer-horizon benchmarks: each subtask is short, self-contained, and follows the same pattern, so the agent never faces the long-horizon planning that degrades performance.
The question is whether agents, given the right scaffolding, can produce \emph{verified} software at a scale that human teams have not achieved.
The scaffolding we propose is \emph{skills}.
A skill is a plain-markdown document, committed to the repository, that reads like a checklist: ``implement the forward and inverse maps, write round-trip tests, draft a paper entry with a proof sketch'' (\Cref{fig:skill-addrule}).
The agent interprets each step using its own capabilities, but the skill ensures nothing is skipped and every step follows the project's conventions.
A \emph{project specification} anchors the system as a single source of truth for architecture and constraints that every skill inherits.
Skills address all three barriers:
\begin{itemize}
  \item \emph{Convention drift} $\to$ skills enforce the same interface, layout, and test pattern at every invocation.
  \item \emph{Knowledge discontinuity} $\to$ domain experts contribute through structured issues; agents handle the implementation.
  \item \emph{Effort exhaustion} $\to$ the routine work---coding, testing, documenting---becomes repeatable without human attention.
\end{itemize}
A domain expert who discovers a new reduction contributes it by filing a structured issue---no code required---and agents handle the implementation, testing, and documentation.
A multilayer verification stack rejects non-conforming code before it enters the repository.

We demonstrate that AI coding agents, constrained by systematic verification, can build mathematical software at a scale that human teams have not achieved.
The evidence is a Rust library of 100+ problem types and 156~reduction rules, built in under three months by a small team and AI agents.
Three ideas make this possible:
\begin{enumerate}
  \item A \emph{skill-based automation pipeline} built on two kinds of skills: \emph{agent skills} that let agents autonomously code, test, and document each reduction, and \emph{advisor skills} that guide users, contributors, and maintainers through the project's conventions (\Cref{sec:skills}).
  \item A \emph{no-code contribution route} that lets domain experts continuously integrate new reduction rules by filing structured issues---no familiarity with the implementation language is needed (\Cref{sec:method}).
  \item A \emph{multilayer verification stack} that ensures correctness at every stage: automated unit tests and round-trip tests validate each reduction programmatically, while domain experts perform end-to-end checks against contributor-specified ground truth (\Cref{sec:verification}).
\end{enumerate}

\subsection{User Interfaces}\label{sec:user-interfaces}

The \texttt{pred} CLI is the primary entry point.
Consider a logistics engineer who models warehouse bin placement as a Maximum Independent Set (MIS) problem---finding the largest set of vertices with no two adjacent---on a conflict graph: vertices are candidate locations, edges connect locations that are too close.

Beyond this workflow, \texttt{pred path} shows the reduction route and per-step overhead between any two problem types, \texttt{pred show} prints a problem's definition and neighboring reductions, and \texttt{pred list} shows all registered problem types with their complexities.
The same functionality is available through the Rust library API.

The build system also produces an auto-compiled PDF manual---formal definitions, reduction theorems with proof sketches, and worked examples---generated from the same source data that powers the CLI.

\subsection{Core Library}\label{sec:graph-structure}

The core crate contains six modules (\Cref{fig:library-architecture}, middle).

\emph{Problem types.}
Every problem type implements a uniform evaluation interface: given any \emph{configuration}---an assignment of values to the problem's decision variables---return an objective value (\Cref{app:architecture}).
Each type defines its own \emph{size measures} and \emph{best-known algorithm complexity}---for example, Maximum Independent Set (MIS) is parameterized by $|V|$ and $|E|$, with a known $O(1.1996^{|V|})$ algorithm~\cite{Xiao2017MIS}.
The uniform evaluation interface enables brute-force solving on small instances, which powers round-trip testing.

\emph{Reduction rules.}
Each rule requires both a forward map and an inverse map in a single implementation---an agent cannot compile a forward reduction without providing the inverse map for solution extraction (\Cref{app:architecture}).
Each rule carries a \emph{reduction overhead}: a multivariate polynomial mapping the source's size measures to the target's.
For example, the Satisfiability $\to$ Maximum Independent Set reduction creates one vertex per literal occurrence in the formula, yielding $O(L)$ vertices and $O(L^2)$ edges where $L$ is the total number of literal occurrences.

\emph{Graph engine.}
Problem types and reduction rules compose into a directed graph.
To route a problem instance to a solver, the engine finds the cheapest reduction path using Dijkstra's algorithm.
The cost function is configurable: minimize the number of reduction steps, or minimize a specific output measure such as the number of variables.

\emph{Symbolic engine.}
Reduction overhead and algorithm complexity are represented as symbolic mathematical expressions, because costs depend on input size, which is unknown until a concrete instance arrives.
The symbolic engine supports three operations.
\emph{Composition} substitutes one edge's output into the next edge's input.
\emph{Comparison} checks whether a composite path dominates a direct rule, for redundancy detection.
\emph{Evaluation} computes numeric cost for a concrete instance size, enabling instance-dependent path search.

\emph{Solvers.}
A brute-force enumerator exhaustively searches small instances---used in round-trip testing where correctness, not speed, matters.
An ILP solver handles larger instances; the backend defaults to the open-source HiGHS~\cite{Huangfu2018HiGHS}, with Coin-CBC~\cite{Forrest2023CoinCBC}, SCIP~\cite{Achterberg2009SCIP}, and others available as drop-in alternatives.
Most problem types have a reduction path to ILP, directly or through intermediate types.

\emph{Example database.}
Canonical instances serve as the single source of truth.
The same instance feeds round-trip tests, JSON fixtures rendered in the PDF manual, and the \texttt{pred create --example} command for interactive exploration.

\subsection{Why Rust?}\label{sec:why-rust}

The choice of implementation language has outsized impact in agentic workflows, because the agent's edit--compile--test loop runs hundreds of times per session.
None of the maintainers had written Rust before this project---the language was chosen in large part for properties that benefit agents.

\emph{Explicit, actionable error messages.}
The Rust compiler produces diagnostics that include the error location, the conflicting types or lifetimes, and often a suggested fix.
Agents parse these messages and resolve errors without human intervention---a property we rely on throughout the pipeline.
Languages with less structured diagnostics (e.g., C++ template errors) would require more agent reasoning per cycle.

\emph{Fast feedback.}
Incremental compilation and the built-in test harness produce results in seconds.
A typical round-trip test compiles and runs in under 3~seconds, enabling agents to iterate rapidly.
Rust's well-known strengths---memory safety eliminating entire bug categories at compile time, high performance, small binary size, and Cargo's integrated toolchain---further reduce the surface area of errors agents must debug.

\section{Skills and Automation}\label{sec:method}

\subsection{Skill Design Principles}\label{sec:skills}
Instead of asking what can be automated, we ask what cannot.
In this project, we found that four responsibilities must remain with humans: \emph{selecting} which reduction to pursue, \emph{verifying} the algorithm's correctness, \emph{constructing} canonical examples, and \emph{authorizing} merges.
Selecting which rules to pursue requires understanding the needs of the community; the human is the information source.
Verifying the algorithm's correctness requires deep reasoning; current models are not yet reliable enough to substitute for humans. Reasoning degrades sharply with problem complexity~\cite{Mirzadeh2025GSMSymbolic,Shojaee2025IllusionOfThinking,Dziri2023FaithFate}. Research-level mathematics remains largely unsolved~\cite{Glazer2024FrontierMath,Paster2025HLE,Merrill2024ExpressivePower}.
Constructing canonical examples requires creative work; the human is the end user and the best judge.
Authorizing merges is security critical; a human with long-term memory must be responsible.

At runtime, the agent reads the skill document and executes each step in sequence.
Through interactive prompts, a skill can pause the agent to solicit feedback from the human, providing context and a recommendation before letting the human decide.
The skills in this project are divided into two types (\Cref{fig:skill-architecture}): \emph{advisor skills} (blue) that focus on utilizing human expertise to make decisions; \emph{agent skills} (white) run fully autonomously.

\textbf{Stage~1: Propose} ( $\to$ \col{Backlog}).
A domain expert clones the repository and runs the \texttt{propose} skill (\Cref{fig:skill-propose}) in Claude Code or a similar agentic coding tool.
The skill conducts a structured brainstorming session---one question at a time---to elicit the contribution's core content: which problem to add, its formal definition, and the reduction algorithm connecting it to existing problems; the expert answers in mathematical language, and no programming is required.
The agent runs a topology analysis to avoid duplicates and to identify high-value gaps in the graph, then pre-validates the draft against Stage~2's quality checks.
The output is a structured GitHub issue filed into \col{Backlog}.

\textbf{Stage~2: Validate} (\col{Backlog} $\to$ \col{Ready}).
A maintainer uses the \texttt{fix-issue} skill to triage each issue in \col{Backlog}.
The skill first runs \texttt{check-issue}, which automatically evaluates the proposal along four dimensions: \emph{usefulness} (does a cheaper path already exist?), \emph{effort} (what is the effort to implement this rule?), \emph{correctness} (do cited references support the claims?), and \emph{writing quality} (are all symbols defined, all examples fully worked?).
It then auto-corrects mechanical failures (undefined symbols, missing metrics) and brainstorms substantive fixes with the maintainer interactively.
Only issues passing all four checks move to \col{Ready}.

\textbf{Stage~3: Implement} (\col{Ready} $\to$ \col{Review Pool}).
An implementation agent executes the \texttt{run-pipeline} skill. It checks the \col{Ready} column, picks an issue with the highest priority (e.g.\ new problem types before new reduction rules, since rules depend on types), claims it by moving it to \col{In Progress}, produces one PR with code implementation and manual writing to resolve the issue, and moves the issue to \col{Review Pool}.
The implementation agent separates two kinds of knowledge.
\emph{Domain skills} (\texttt{add-model}, \texttt{add-rule}) encode what makes a reduction correct: a correctness argument, a size overhead expression, and a worked example must all exist \emph{before any code is written} (\Cref{fig:skill-addrule}).
This is knowledge the agent cannot infer from existing code---it must come from the issue's mathematical content.
\emph{Engineering skills} (\texttt{issue-to-pr}) encode how to produce a clean PR: isolated worktrees, implementation plans, parallel sub-agents for independent tasks, and a detailed summary---all fully automatable.\footnote{The engineering workflow builds on Superpowers~\cite{Obra2025Superpowers}, an agentic skills framework for structured development workflows.}

\textbf{Stage~4: Review} (\col{Review Pool} $\to$ \col{Final Review}).
A review agent executes the \texttt{review-pipeline} skill. It claims an issue from \col{Review Pool} by moving it to \col{Under Review}, dispatches three read-only sub-agents in fresh context windows---structural completeness, code quality, and agentic feature testing (AI agents role-playing as end users; \Cref{sec:verification})---and posts a review report to the PR comment and moves the issue to \col{Final Review}.

\textbf{Stage~5: Merge} (\col{Final Review} $\to$ \col{Done}).
A maintainer uses the \texttt{final-review} skill to merge or hold the PR.
The skill picks a PR from \col{Final Review}, checks the report from the review agent, automatically fixes mechanical issues and presents uncertain fixes to the maintainer for decision.
It then walks the maintainer through the agentic verdict, assesses usefulness, safety, and completeness, and either merges (moves the issue to \col{Done}) or holds the PR with a reason (moves the issue to \col{OnHold}).

\textbf{Stage~6: Verify.}
The maintainer invites the domain expert who filed the original issue to a community call,
walks through the PDF manual and CLI interactively.
This closing-the-loop step catches errors invisible to automated tests: subtle misinterpretations of the problem statement or proof arguments.

Stages~3--4 (implement and review) are fully automatable and run in \emph{headless mode}---no human interaction---via OpenAI Codex~\cite{OpenAI2025Codex} (\texttt{gpt-5.4}) with highest thinking effort.
The remaining stages use Claude Code~\cite{Anthropic2025ClaudeCode} (\texttt{opus-4.6}) for interactive sessions with the maintainer and domain expert.
Anecdotally, we found \texttt{gpt-5.4} more reliable for code implementation and \texttt{opus-4.6} stronger in interactive dialogue, though we did not evaluate this systematically.

\subsection{Correctness by Construction}\label{sec:verification}

Beyond unit tests and manual verification, we adopt the following two mechanisms to ensure correctness.

\textbf{Issue check (Stage 2).}
Correctness begins with the issue provided by the domain expert.
As shown in \Cref{fig:roundtrip-pipeline}, the structured issue serves as both the specification and the acceptance criterion: it defines the reduction's mathematical content, provides the canonical example that seeds all tests.
The issue is validated along the four dimensions described in Stage~2 (\Cref{sec:pipeline-overview}).
Invalid proposals are rejected before implementation effort is spent.
The issue's canonical example---a small, fully worked instance with known optimal solution---later seeds the round-trip test, the PDF manual's visual diagram, and the CLI's \texttt{-{}-example} mode.
The pipeline is designed so that the mathematical intent propagates through all downstream artifacts; the contributor's final review confirms this.

\textbf{Agentic feature tests (Stage 4).}
In this project, agentic feature tests compressed the usual feedback loop.
Libraries normally stabilize through community use, but a niche library cannot wait for that community to form.
Agentic feature tests approximate early-adopter feedback by simulating realistic use scenarios before real users arrive.

The mechanism has three phases.
First, the main agent launches a sub-agent in a \emph{fresh context window}---one that has never seen the implementation---and injects an \emph{agent profile}: a structured persona with domain expertise, testing habits, and a concrete use case (\Cref{fig:agent-profile}).
The fresh context is essential; a sub-agent that has just written the code cannot objectively test it.
The main agent's framing could still introduce bias, but the sub-agent discovers the library independently through its documentation.

\section{Related Work}\label{sec:related}

\textbf{Reduction libraries and QUBO tools.}
Several tools convert optimization problems into a single target formulation.
PyQUBO~\cite{Zaman2021PyQUBO}, qubovert~\cite{Iosue2022qubovert}, ToQUBO.jl~\cite{Monteiro2022ToQUBO}, and D-Wave's Ocean SDK~\cite{DWaveOcean2023} target QUBO/Ising models but provide no inter-problem reductions and no inverse maps for solution extraction.
Lucas~\cite{lucas2014} catalogued Ising formulations for Karp's 21~problems; the result is a reference, not composable code.
The Karp DSL~\cite{Zhang2022Karp} and REDNP~\cite{Creus2014REDNP} provide languages for writing and testing reductions, but are pedagogical tools with no solver integration.
ProblemReductions.jl~\cite{Liu2024ProblemReductions}, our earlier Julia prototype, implements 18~problem types with a reduction graph structure; its limited growth motivated the present work.
Our library differs from all of these in three ways: it connects \emph{many} source types to \emph{many} target types through a graph with composable overhead, it requires inverse maps for solution recovery, and it is maintained by AI agents rather than by hand.

\textbf{AI coding agents.}
SWE-agent~\cite{Yang2024SWEagent}, Devin~\cite{Wu2024Devin}, OpenHands~\cite{Wang2024OpenHands}, and Claude Code~\cite{Anthropic2025ClaudeCode} resolve isolated GitHub issues with increasing reliability~\cite{Xia2025LiveSWEagent}, but longer-horizon benchmarks reveal a capability cliff~\cite{Thai2025SWEEVO, Deng2025SWEBenchPro}, and specification inference remains the central difficulty~\cite{Roychoudhury2025AgenticAI}.
These systems operate on arbitrary codebases with minimal structure.
Our approach exploits the assembly-problem structure---homogeneous, verifiable subtasks---so that each agent invocation is short and self-contained, sidestepping the long-horizon planning that degrades general-purpose agents.

\textbf{Structured agent workflows.}
Chain-of-thought prompting~\cite{Wei2022CoT}, ReAct~\cite{Yao2023ReAct}, and tool-use frameworks decompose agent reasoning into steps, but the decomposition is generated per invocation and is not reusable across tasks.
Our skills are closer to stored procedures: persistent, version-controlled documents that encode domain-specific workflows and are inherited by every agent session.
Superpowers~\cite{Obra2025Superpowers} provides a general-purpose skills framework; we build on it for engineering workflows while adding domain-specific skills for reductions.

\section{Discussion and Conclusion}\label{sec:conclusion}

\subsection{Discussion}

The methodology generalizes to what we call a \emph{gallery library}---a collection of homogeneous, independently verifiable implementations with uniform interfaces.
An automatic differentiation (AD) engine is a natural example: each differentiation rule transforms one operation into its derivative, and finite-difference checks play an analogous role---approximate but generic---verifying correctness automatically, without a problem-specific oracle.
The key requirement is a cheap, generic verification mechanism that applies uniformly across all entries.
The cost of entry is skill engineering: the 18~skills in this project represent substantial upfront investment, and each new domain requires its own set.
The methodology does \emph{not} generalize to heterogeneous tasks, where each issue is structurally unique and resists skill-based decomposition.

The library itself can serve as a \emph{tool} for AI agents: an agent facing an optimization problem can invoke the \texttt{pred} CLI to route the problem through the reduction graph to a solver, without knowing which solver fits or how to formulate the reduction.
The reduction graph also serves as a \emph{solver-agnostic compilation layer}: adding a single edge from a hardware platform's native formulation (e.g., Maximum Independent Set, which Rydberg-atom quantum hardware solves natively) routes all supported problem types to that device.

Three directions extend this work.
First, composing the library with \emph{automated reduction discovery} via evolutionary search~\cite{Novikov2025AlphaEvolve}.
Second, replacing asymptotic overhead with \emph{fine-grained cost prediction}---using extracted problem features to estimate concrete reduction cost per instance, enabling adaptive solver routing.
Third, extending the graph beyond NP to \emph{other complexity classes} (approximation hierarchies such as APX and PTAS, counting problems in \#P), preserving hardness guarantees across reductions.

\subsection{Conclusion}

We have introduced \emph{assembly problems}---software projects whose scale exceeds human capacity but whose homogeneous, verifiable structure makes them amenable to agent execution constrained by systematic verification.
NP-hard problem reductions are a concrete instance: in under three months, a small team and AI agents produced a verified reduction graph connecting 100+ problem types to specialized solvers.

Three structural barriers---convention drift, effort exhaustion, and knowledge discontinuity---make assembly problems impractical for human teams at this scale and pace.
Verification ensures that agents do not deviate unchecked from contributor-specified ground truth, and skills encode workflow knowledge as reusable documents, lowering the contribution barrier from ``knows the programming language'' to ``knows the mathematics.''
With 100+ problem variants already connected, the graph routes supported problems to solvers along cheapest paths.

\section{System Architecture}\label{app:architecture}

The library's type system reduces the space of possible agent errors by making incorrect code fail to compile.
This appendix describes the key design decisions: the aggregation-based problem representation, the solver priority system, and the variant registry (\Cref{fig:architecture}).

\begin{figure}[t]
  \centering
  \includegraphics[width=\columnwidth]{figures/architecture.pdf}
  \caption{Trait hierarchy and compile-time validation.
    The \texttt{Problem} trait defines a universal evaluation interface; \texttt{ReduceTo<T>} requires both forward and inverse maps; procedural macros validate overhead expressions and variant registrations at compile time.}
  \label{fig:architecture}
\end{figure}

\paragraph{The Problem trait and aggregation.}
Every problem type implements \texttt{Problem}, which requires: a constant name, an associated \texttt{Value} type, a method returning the configuration space dimensions, an \texttt{evaluate()} method that maps any configuration to a value, and variant metadata.
The \texttt{Value} type is an \emph{aggregation wrapper} that determines how individual evaluations combine across the configuration space.
Five wrappers cover the standard problem classes:
\texttt{Max<W>} and \texttt{Min<W>} for NP optimization problems (e.g., Maximum Independent Set returns \texttt{Max<i32>});
\texttt{Or} for NP decision problems (e.g., $k$-Satisfiability returns \texttt{Or});
\texttt{Sum<W>} for \#P counting problems; and
\texttt{And} for co-NP decision problems.
These wrappers anticipate future extensions beyond NP (\Cref{sec:conclusion}) and cost little to include.
Each wrapper implements an \texttt{Aggregate} trait with an \texttt{identity()} and a \texttt{combine()} method, so a brute-force solver reduces to a single fold over the configuration space: \texttt{configs.map(evaluate).fold(identity, combine)}.
The same trait marks which wrappers support \emph{witness recovery}: \texttt{Max}, \texttt{Min}, and \texttt{Or} can track which configuration achieved the optimal value; \texttt{Sum} and \texttt{And} cannot.
This distinction propagates to the reduction traits.

\paragraph{Two-track reduction traits.}
The type system enforces a split based on witness capability.
\texttt{ReduceTo<T>} requires a \texttt{ReductionResult} that bundles \texttt{target\_problem()} and \texttt{extract\_solution()}---a map from a target \emph{configuration} back to a source configuration.
An agent cannot compile a forward reduction without providing the inverse map; incomplete reductions are compile errors.
\texttt{ReduceToAggregate<T>} requires an \texttt{AggregateReductionResult} with \texttt{extract\_value()} instead---a map from a target \emph{value} back to a source value, with no witness recovered.
For subtype edges (e.g., MIS on a king's subgraph to MIS on a general graph), a generic \texttt{ReductionAutoCast} implements both traits with an identity solution map.
At runtime, type-erased blanket implementations (\texttt{DynReductionResult}) let the graph engine chain reductions along Dijkstra-discovered paths without knowing the concrete types at compile time.
Each reduction also carries a compile-time-validated overhead: the \texttt{\#[reduction(overhead = \{...\})]} procedural macro attaches symbolic size expressions, and variable names are checked against getter methods on the source type---a typo causes a compile error, not a silent bug.

\paragraph{Solver priority.}
When the user invokes \texttt{pred solve}, the system selects a solver in priority order.
First, if the problem has a dedicated solver that exploits its structure (currently six problem types), the system uses it.
Second, if a witness-capable reduction path to Integer Linear Programming (ILP) exists, the system reduces along the cheapest path (by Dijkstra) and solves with HiGHS.
Third, the brute-force enumerator is always available: it folds over all configurations using the aggregation interface.
Brute force remains essential for round-trip testing, where correctness matters more than speed.

\paragraph{Variant registry.}
A single problem type like Maximum Independent Set admits multiple \emph{variants}: parameterizations by graph type (\texttt{SimpleGraph}, \texttt{KingsSubgraph}, \texttt{UnitDiskGraph}, etc.) and weight type (unit, \texttt{i32}, \texttt{f64}).
Each variant may have a better complexity bound---MIS on king's subgraphs runs in $O(2^{\sqrt{|V|}})$ versus $O(1.1996^{|V|})$ on general graphs.
The \texttt{declare\_variants!} macro registers each concrete instantiation with its best-known complexity, a factory for JSON deserialization, and a solve function.
All variable names in complexity expressions are validated at compile time against the problem's getter methods.
This registry is what produces the registered variants from 100+ base problem types.
